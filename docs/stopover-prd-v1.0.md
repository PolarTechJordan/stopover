## 1. Problem Statement

### The Problem
中转旅客（layover / stopover，时长 6–48 小时）在机场停留期间面临三类核心焦虑：

1. **时间焦虑**：6–48 小时不知如何安排，候机厅等待体验差
2. **行李焦虑**：拖箱出行受限，机场外活动意愿被压制
3. **体验焦虑**：陌生中转地缺攻略、缺信任，独自探索风险高

与此同时，机场生态服务商（休息室、餐饮、接送机、快速安检、eSIM 等）的中转场景长尾服务渗透率低——现有休息室产品仅覆盖旅客停留 2–3 小时中的核心时段，剩余大量时间未被有效货币化。

### Evidence
- **竞品参照**：
  - 新加坡樟宜机场、卡塔尔多哈哈马德机场：机场内观光路线（花园、滑道、瀑布）已形成标志性中转体验
  - 土耳其航空：8/10/12 点固定时段出发的免费中转城市游（限制乘坐土航联程）
  - 携程 × 首都机场：上午长城 / 下午前门"白天纯玩"路线
- **市场缺口**：上述产品或绑定单一航司、或仅覆盖机场内，缺乏"任意中转旅客 + 机场内外全链路"的服务整合
- **内部数据机会**（待补充）：休息室用户中转时长分布、停留 > 6h 用户比例、增值服务购买转化率

### Why Now?
1. 国际航线恢复，中转旅客流量增长
2. 高净值用户对"机场生态一站式"服务预期提升
3. 套餐化打包（休息室+安检+eSIM+接送机+酒店）已成行业标配雏形，单点产品已不足以构筑壁垒

## 2. Non-Goals / Out of Scope

**第一版不做：**

- ❌ **不绑定特定航司**：不复制"土耳其航空仅限土航联程"模式，对所有持有中转航段的用户开放购买
- ❌ **不做机场外长线深度游**：仅覆盖中转时段 6–48h，跨城多日游不在范围
- ❌ **不做签证 / 入境服务**：免签或可落地签国家才纳入
- ❌ **不做 GDS 直连动态打包**：第一版不与航司动态打包，仅做自有套餐 SKU
- ❌ **不做 B2B 旅行社分销渠道**：第一版聚焦 C 端用户，B2B 渠道留待 V2
- ❌ **不做全机场覆盖**：第一版选择 3–5 个标志性中转枢纽做 PoC（如：新加坡樟宜、伊斯坦布尔 ATQ、多哈 DXB 等）

## 3. Goals & Success Metrics

### Primary Goal
**为中转 6–48 小时的旅客提供"机场生态一站式 + 城市微游"服务包，把单次中转的客单价从休息室单点购买提升到套餐化消费。**

### Success Metrics

| 指标 | 当前（基线） | 目标 | 时间窗 |
|------|------------|------|--------|
| 套餐渗透率（购买套餐/进入中转场景用户） | TBD（基线待定） | 8% | 上市后 90 天 |
| 单中转客单价（ARPL） | 仅休息室约 ¥200 | ¥450+ | 上市后 90 天 |
| 复购 / 转介绍率 | TBD | 15% | 上市后 180 天 |
| 行李托管转化率（购买套餐中含行李服务的比例） | N/A | 30% | 上市后 90 天 |
| 用户 NPS | TBD | ≥ 50 | 上市后 90 天 |
| 套餐履约零事故率 | — | 100%（漏接/误机赔付） | 全周期 |

### Hypothesis
> 我们相信：通过"休息室作信任锚点 + 行李托管消除出行阻力 + 标准化城市微游消除决策成本"的三段式套餐化设计，能让原本只购买休息室的用户多付 ¥250+，且不增加其时间/行李焦虑。

## 4. Target Users & Personas

### Primary Persona：探索型中转商旅 / 高净值自由行旅客
- **角色**：30–50 岁，常飞国际线，单段中转 8–24h，年出行 6+ 次
- **目标**：用中转时间干点有质量的事（吃点好的、看看风景）而非纯等待
- **痛点**：不愿拖箱出机场、缺当地攻略、英语或当地语言弱、怕误机
- **决策风格**：对价格不敏感，对**时间确定性 + 一站式便利**最敏感
- **触点**：航司 App、OTA、机场休息室

### Secondary Persona：探亲 / 留学往返中转的家庭旅客
- **角色**：35–55 岁，带 1–2 个孩子，停留 12–36h
- **目标**：让孩子有地方玩、有地方睡、不折腾
- **痛点**：孩子精力有限、酒店/钟点房不熟悉、行李多
- **差异点**：对**酒店/钟点房 + 儿童餐 + 接送**需求强烈，纯观光意愿低

### Tertiary Persona：红眼航班 / 长途中转休息需求旅客
- **角色**：任何年龄段，中转 6–12h，主要是睡觉和洗漱
- **目标**：高效补眠后精神登机
- **痛点**：休息椅不舒服、淋浴稀缺、行李安全
- **差异点**：对**睡眠舱 / 钟点房 + 淋浴 + 行李寄存**需求强，不关心城市游

### User Journey Context
```
[出发] → [中转地到达] → [提取/不提取托运行李] → [中转柜台办理]
   → [选择本产品的服务套餐] → [行李托管 → 休息室/酒店/城市游] → [回到机场] → [取回行李] → [安检登机]
```

## 5. Proposed Solution

### Solution Overview
构建"**以机场休息室为信任锚 + 行李全托管 + 模块化城市服务**"的中转服务套餐产品。用户在中转柜台/线上一次性购买套餐后，**行李由我们负责转运**，人可自由选择"在休息室"、"去城市微游"、"去酒店钟点房"中的一种或多种组合。

### Feature 1：中转服务套餐（MVP 主轴）

**Priority**: Must-Have

**User Story**:
> As a 中转旅客，我想要**一次性购买包含行李托管 + 休息室 + 自选增值项的套餐**，so that 我不用分别决策每项服务、降低时间与决策焦虑。

**Description**:
设计 3 档基础套餐 + N 项增值 SKU：

| 套餐 | 包含 | 适用场景 | 定价区间（待定） |
|------|------|----------|----------------|
| **轻享包** | 休息室 3h + 行李寄存 + 快速安检 | 6–8h 中转，纯休息 | ¥260–320 |
| **微游包** | 休息室 2h + 行李托管 + 城市半日游（4h） + 接送 | 10–18h 中转，白天 | ¥680–880 |
| **过夜包** | 休息室 1h + 行李托管 + 合作酒店/钟点房 + 接送 | 12–36h 跨夜 | ¥780–1200 |

**Acceptance Criteria**:
```gherkin
Scenario: 套餐下单
  Given 用户在 App/柜台选择中转机场与时段
  When 用户选定套餐与增值项（如：升级酒店、加购 eSIM）
  Then 系统在 60 秒内生成订单并扣款
  And 系统下发电子凭证（含 QR 码 + 服务触发时点）

Scenario: 行李托管
  Given 用户已下单"含行李托管"套餐
  When 用户在中转柜台交付随身行李
  Then 我们在 15 分钟内将行李转运至合作贵宾厅/酒店
  And 用户收到"行李已到 X"推送
  And 行礼包 RFID 绑定用户订单

Scenario: 误机保障
  Given 用户购买了"微游包"或"过夜包"
  If 城市游/酒店环节导致用户未能在登机前 60 分钟到达安检口
  Then 系统自动启动误机保障：改签协助 + 酒店+餐食赔付
  And 客服 5 分钟内介入
```

### Feature 2：行李全托管服务

**Priority**: Must-Have

**User Story**:
> As a 中转旅客，我想要**下机后就把行李交给你们**，so that 我可以轻装去任何地方（休息室、城市、酒店），不用担心误机或行李丢失。

**Description**:
- 中转柜台收件 → 专车/专道转运到贵宾厅或合作酒店
- RFID + 拍照登记 + 用户签字
- 离开机场时在贵宾厅或登机口取回
- 含 ¥5000/件行李险

**Acceptance Criteria**:
```gherkin
Scenario: 行李收取
  Given 用户到达中转柜台并出示订单 QR
  When 工作人员扫码确认订单包含行李托管
  Then 行李在 15 分钟内被收走并贴上 RFID 标签
  And 用户手机收到"行李已收"通知（含行李照片 + RFID 编号）

Scenario: 行李送回
  Given 用户的接驳航班将在 90 分钟内起飞
  When 用户在贵宾厅或指定行李归还点
  Then 用户出示订单 QR 完成签收
  And 若超过 90 分钟仍未签收，系统自动报警并优先派送

Scenario: 行李异常
  If 行李在托管期间发生丢失、损坏
  Then 系统按保险条款自动赔付（最高 ¥5000/件）
  And 用户在 30 分钟内拿到临时生活用品补偿包
```

### Feature 3：标准化城市微游（半日 / 一日）

**Priority**: Must-Have

**User Story**:
> As a 中转旅客，我想要**不用做攻略的 4/6/8 小时城市游**，so that 我能在陌生城市看到最经典的几个点，且不会因为语言/路线而误机。

**Description**:
- 借鉴土耳其航空 8/10/12 点固定出发 + 携程 × 首都机场"上午长城 / 下午前门"模型
- 每个中转枢纽预设 **3–5 条**"固定时段 + 固定路线"产品：

| 路线类型 | 出发时段 | 行程 | 价格 |
|---------|---------|------|------|
| 经典 4h | 上午 9:00 / 下午 14:00 | 3 个核心点位 + 1 餐 | ¥380 |
| 深度 6h | 上午 9:00 | 5 个点位 + 1 餐 | ¥580 |
| 一日 8h | 上午 8:30 | 含早茶/午餐、博物馆、地标 | ¥780 |

- 含专车 + 中文/英文向导 + 误机保障
- **路线设计原则**：景点相对集中、避开高峰、确保 60min 前回到机场

**Acceptance Criteria**:
```gherkin
Scenario: 路线预订
  Given 用户选择"微游包"且停留 ≥ 10h
  When 用户在 App 选择出发时段（9:00 / 14:00）
  Then 系统显示该时段可预订的 3 条路线
  And 路线详情包含：景点列表、用时、车程、午餐、回到机场时间

Scenario: 误时保护
  Given 城市游进行中
  If 当前时间超过"返程截止时间 − 30 分钟"且游客仍在景点
  Then 向导立即启动"快速返程流程"：专车+VIP 安检
  And 系统通知机场贵宾厅预留位置
```

### Feature 4：模块化增值项（eSIM / 接送机 / 酒店钟点房 / 淋浴）

**Priority**: Should-Have

**User Story**:
> As a 中转旅客，我想要**根据自己需要加购单项服务**（如只要 eSIM 或只要钟点房），so that 我的中转方案可以量身定做。

**Description**:
- eSIM：中转地 4G/5G，按天/按流量包
- 接送机：机场 ↔ 酒店 / 城市
- 酒店钟点房：合作酒店 4h/6h 时段
- 淋浴 / 睡眠舱：机场内单点
- 机场餐饮券：与机场商户联合的折扣券

**Acceptance Criteria**：
```gherkin
Scenario: 增值项加购
  Given 用户已选基础套餐
  When 用户在"加购"页选择 eSIM/接送/钟点房
  Then 系统按所选加购项计算总价
  And 单项加购也可独立售卖（不强制买基础套餐）

Scenario: 联合权益核销
  Given 用户购买了含机场餐饮券的套餐
  When 用户在合作商户出示 QR
  Then 商户扫码后立享折扣
  And 系统核销后推送"已使用"通知
```

### Feature 5：包车 / 私人定制游（高端 SKU）

**Priority**: Could-Have

**User Story**:
> As a 到达时间晚但仍想看城市的中转旅客，我想要**一辆专车 + 司机陪游**，so that 我可以根据自己状态灵活调整路线。

**Description**：
- 时长 3h / 5h / 8h 三档
- 中文/英文司机兼向导
- 路线用户临时调整（但需不晚于返程截止时间）

**Acceptance Criteria**：
```gherkin
Scenario: 包车预订
  Given 用户停留 6–12h 且非传统作息（红眼到达）
  When 用户选择包车 4h
  Then 系统指派车型 + 司机
  And 司机按用户到达时间接机
```

## 6. Technical Approach & Non-Functional Requirements

### Architecture Overview
- 入口：App 内"中转服务"频道 + 机场中转柜台扫码进入
- 后端：服务编排（休息室系统 + 行李系统 + 城市游供应商 + 酒店 PMS + eSIM 平台 + 安检系统）通过统一订单中台串联
- 关键模块：
  - **订单中台**：套餐 SKU、订单状态机、履约追踪
  - **行李调度**：RFID + IoT 定位 + 异常告警
  - **城市游调度**：供应商路由 + 时段库存 + 误时保护
  - **误机保障引擎**：航班动态接入（FlightStats/航旅纵横）+ 触发式赔付

### Technical Requirements

#### 性能
- 订单生成 P95 < 1s
- 行李状态推送延迟 < 30s
- 城市游时段库存查询 P95 < 500ms

#### 安全 / 合规
- 护照 / 签证信息：脱敏存储，**仅用于入境核验**，不进入营销画像
- 行李 RFID 数据 90 天后自动清除
- 符合当地机场安检与海关对接要求

#### 可靠性
- 服务可用性 ≥ 99.5%
- 误机保障引擎：航班动态延迟 < 2min
- RTO：单点服务故障 < 30min 切到人工兜底

### Data Requirements

**New Data Entities**:
```
Entity: StopoverOrder
- order_id: string (PK)
- user_id: string (FK)
- layover_airport: string (IATA)
- arrival_flight: string
- departure_flight: string
- package_sku: enum (light/micro/overnight)
- addons: list[string]
- status: enum (paid/in_progress/completed/refunded)
- total_amount: decimal
- created_at, updated_at: timestamp

Entity: BaggageTracking
- tracking_id: string (PK)
- order_id: string (FK)
- rfid_tag: string
- pickup_location: string
- current_location: string
- status: enum (received/in_transit/at_lounge/delivered/lost)
- photos: list[url]
- last_updated: timestamp

Entity: CityTourBooking
- tour_id: string (PK)
- order_id: string (FK)
- route_id: string
- departure_time: time
- return_buffer_min: int (默认 90)
- guide_id: string
- vehicle_id: string
- status: enum
```

**State Transitions（核心订单）**:
```
[Paid] → [BaggageReceived] → [InLounge|InCityTour|AtHotel] → [BaggageReturned] → [Boarded] → [Completed]
                                                                       ↘ [MissedFlight:RefundFlow]
```

### Integrations

| 系统 | 用途 | 接入方式 |
|------|------|---------|
| 休息室系统 | 凭证核销 | API |
| 行李系统 | RFID 调度 | IoT + Webhook |
| 航班动态（FlightStats / 航旅纵横） | 误机保护 | 推送 |
| 机场安检 | 快速通道核销 | API/扫码 |
| 酒店 PMS | 钟点房库存 | API |
| eSIM 平台 | 流量包下发 | API |
| 城市游供应商 | 订单路由 | Open API |

## 7. Dependencies & Constraints

### External Dependencies
- [ ] 机场方准入协议（行李托管区域、贵宾厅联营）— Owner: BD
- [ ] 城市游供应商签约（每个枢纽至少 2 家以做备份）— Owner: 采购
- [ ] 酒店 / 钟点房合作 PMS 接入 — Owner: 酒店 BD
- [ ] 机场餐饮 / 购物商户联合权益谈判 — Owner: BD
- [ ] 当地导游/向导资源（中文/英文）— Owner: 当地运营

### Technical Constraints
- 第一版仅支持**已免签/落地签**中转地，签证服务不接入
- 行礼包必须可兼容**最大 28 寸行李 + 1 件手提**
- 城市游需符合当地旅游合规要求（如：泰国需持 TAT 牌照的供应商）

### Resource Constraints
- 预算：MVP 阶段需在 3–5 个枢纽做 PoC，单枢纽启动成本约 ¥80–150 万
- 时间：建议 Q3 完成产品设计 + 1 个枢纽 PoC，Q4 灰度 3 枢纽
- 团队：需要 BD + 当地运营 + 客服 7×24 三线团队

## 8. Risks & Assumptions

### Assumptions

| 假设 | 若不成立的影响 | 验证方式 |
|------|---------------|---------|
| 中转用户愿意为"省心"多付 ¥200–400 | 套餐渗透率 < 3%，ROI 不达标 | 落地页 A/B 测试 + 早期 50 单用户访谈 |
| 行李托管的安全与隐私可被用户信任 | 转化率 < 10% | 包装保险 + 视频记录，验证签约率 |
| 机场方愿意开放行李柜台合作 | BD 周期延长 3–6 个月 | Q3 前与至少 2 家枢纽签 MOU |
| 固定时段出发模型能满足大部分用户 | 部分时段满座、部分时段空 | 实时库存监控 + 弹性加车 |
| 6h 以下中转不是核心场景，6h 以上的占比可观 | 用户盘子比预期小 | 与机场方数据交叉验证 |

### Risks

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 行李丢失/错送 | 中 | 高 | RFID + 保险 + 赔付 SOP + 用户即时通知 |
| 误机事件（套餐口碑风险） | 中 | 高 | 误机保障金 + 改签绿色通道 + 公关预案 |
| 当地供应商服务质量不稳 | 中 | 中 | 双供应商备份 + 神秘客 + 实时评分 |
| 当地法规变化（如：突然收紧免签） | 低 | 高 | 动态下架中转地 + 用户通知 |
| 跨部门协调（航司/机场/海关） | 高 | 中 | 早期拉齐 MOU + 单点负责人制度 |
